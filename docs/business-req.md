## 1. 業務背景與目標

> **⚠️ 注意：** 原始輸入文件僅包含技術規格，缺乏業務背景描述。以下為根據「產品銷售應用」名稱所做的**最小合理推斷**，需業務方確認。

本系統旨在提供一個**產品銷售平台**，支援商品瀏覽、訂單建立、付款處理及訂單履行等核心銷售流程。系統需具備高可靠性，並透過分散式交易機制確保資料一致性。

## 2. 使用者故事（User Stories）

> 角色定義待 Q1、Q2 確認後可能需要調整。

### 2.1 商品目錄管理

---

**US-001 — 瀏覽商品列表**

```
身為一位 訪客／會員
我希望 能夠瀏覽所有上架中的商品列表
以便 找到我想購買的商品
```

**驗收條件：**
- 商品列表僅顯示狀態為「上架」的商品
- 每筆商品顯示：名稱、售價（TWD）、簡短描述、庫存狀態（*待 Q5 確認*）
- 支援分頁瀏覽

---

**US-002 — 查看商品詳情**

```
身為一位 訪客／會員
我希望 能夠查看單一商品的詳細資訊
以便 在購買前了解商品規格與價格
```

**驗收條件：**
- 顯示商品完整資訊：名稱、售價（TWD）、詳細描述、商品圖片
- 若商品不存在或已下架，顯示適當的錯誤提示
- 庫存數量顯示規則待 Q5 確認

---

**US-003 — 上架商品（管理員）**

```
身為一位 管理員
我希望 能夠新增並上架商品
以便 讓顧客可以購買新商品
```

**驗收條件：**
- 必填欄位：商品名稱、售價（TWD）、商品描述
- 售價必須大於 TWD 0
- 商品建立後預設狀態為「草稿」，需明確執行「上架」操作後才對外顯示
- 商品名稱不可重複（*待業務確認是否為必要規則*）

---

**US-004 — 下架商品（管理員）**

```
身為一位 管理員
我希望 能夠將商品下架
以便 停止販售特定商品
```

**驗收條件：**
- 下架後商品不再顯示於商品列表
- 若該商品存在「處理中」的訂單，下架行為規則待 Q8 確認

---

### 2.2 購物車

> **⚠️ Q13（新增）：** 是否需要購物車功能？或直接進行單品結帳？以下故事假設需要購物車，待確認。

---

**US-005 — 加入購物車**

```
身為一位 會員
我希望 能夠將商品加入購物車
以便 一次結帳多項商品
```

**驗收條件：**
- 可指定商品數量，數量最小為 1
- 同一商品重複加入時，累加數量
- 購物車金額小計即時以 TWD 顯示

---

**US-006 — 修改購物車內容**

```
身為一位 會員
我希望 能夠修改購物車中的商品數量或移除商品
以便 在結帳前調整我的購買清單
```

**驗收條件：**
- 可調整商品數量（最小為 1）
- 可移除單一商品
- 可清空整個購物車
- 修改後購物車總金額（TWD）即時更新

---

### 2.3 訂單管理

---

**US-007 — 建立訂單**

```
身為一位 會員
我希望 能夠將購物車內容轉換為訂單
以便 完成購買流程
```

**驗收條件：**
- 訂單建立時鎖定商品售價（TWD），後續商品價格異動不影響已建立的訂單
- 訂單建立後狀態為「待付款」
- 庫存扣減規則待 Q5 確認
- 系統產生唯一訂單編號

---

**US-008 — 查詢訂單狀態**

```
身為一位 會員
我希望 能夠查詢我的訂單狀態
以便 了解訂單的處理進度
```

**驗收條件：**
- 可查詢個人所有歷史訂單列表
- 可查詢單筆訂單詳情，包含：訂單編號、商品明細、金額（TWD）、目前狀態
- 僅能查詢自己的訂單，不可查詢他人訂單

---

**US-009 — 取消訂單**

```
身為一位 會員
我希望 能夠取消尚未付款的訂單
以便 在改變心意時不被強制完成購買
```

**驗收條件：**
- 僅「待付款」狀態的訂單可由會員主動取消
- 取消後訂單狀態變更為「已取消」
- 庫存回補規則待 Q5 確認
- 其他狀態下的取消條件待 Q8 確認

---

### 2.4 付款處理

---

**US-010 — 進行付款**

```
身為一位 會員
我希望 能夠對「待付款」的訂單進行付款
以便 完成購買並等待商品出貨
```

**驗收條件：**
- 付款金額必須與訂單金額（TWD）一致
- 付款成功後訂單狀態變更為「已付款」
- 付款失敗時訂單維持「待付款」狀態，並提供明確的失敗原因
- 支援的付款方式待 Q4 確認
- 付款流程需透過 SAGA 模式確保分散式一致性（補償交易細節待 Q4、Q5 確認）

---

**US-011 — 付款失敗補償**

```
身為一位 系統
我希望 在付款流程中任一步驟失敗時自動執行補償交易
以便 確保訂單、庫存、付款狀態的資料一致性
```

**驗收條件：**
- 付款服務失敗時，已扣減的庫存需自動回補（*待 Q5 確認*）
- 補償交易執行結果需記錄於系統日誌
- 補償失敗時需觸發人工介入告警機制（告警管道待 Q10 確認）

## 4. 詞彙表（Glossary）

| 術語 | 定義 | 備註 |
|------|------|------|
| **商品（Product）** | 可供銷售的品項，具有名稱、描述、售價（TWD）及上下架狀態 | 商品類型待 Q1 確認 |
| **商品目錄（Product Catalog）** | 所有商品的集合，包含上架與下架商品 | |
| **售價（Selling Price）** | 商品對外公開的銷售金額，單位為 TWD | |
| **訂單（Order）** | 會員確認購買意圖後由系統建立的交易記錄，包含商品快照與金額（TWD） | |
| **訂單編號（Order ID）** | 系統自動產生的唯一訂單識別碼 | |
| **價格快照（Price Snapshot）** | 訂單建立當下鎖定的商品售價（TWD），不受後續價格異動影響 | |
| **購物車（Cart）** | 會員暫存欲購買商品的容器，結帳前可自由修改 | 待 Q13 確認是否需要 |
| **付款（Payment）** | 會員針對訂單進行的金額（TWD）支付行為 | 付款方式待 Q4 確認 |
| **訂單狀態（Order Status）** | 訂單在生命週期中的所在階段 | 見下方狀態機說明 |
| **待付款（Pending Payment）** | 訂單已建立但尚未完成付款的狀態 | |
| **已付款（Paid）** | 付款成功確認後的訂單狀態 | |
| **已取消（Cancelled）** | 訂單被取消後的終止狀態 | |
| **上架（Published）** | 商品對外公開可購買的狀態 | |
| **下架（Unpublished）** | 商品不對外顯示、不可購買的狀態 | |
| **草稿（Draft）** | 商品建立後尚未上架的初始狀態 | |
| **管理員（Admin）** | 具有商品管理權限的系統操作者 | 角色權限細節待確認 |
| **會員（Member）** | 已完成註冊並登入的系統使用者 | 會員註冊流程待確認 |
| **訪客（Guest）** | 未登入的系統使用者，僅具有瀏覽權限 | |
| **SAGA 模式（SAGA Pattern）** | 一種分散式交易管理模式，透過一系列本地交易與補償交易確保最終一致性 | 本系統採用 Apache Camel 實作 |
| **補償交易（Compensating Transaction）** | 當 SAGA 流程中某步驟失敗時，用於回滾前序步驟影響的反向操作 | |
| **庫存（Inventory）** | 商品的可售數量 | 待 Q5 確認是否納入範疇 |

### 4.1 訂單狀態機（Order State Machine）

```
[草稿] ──上架操作──▶ [上架]
                         │
                         ▼
[訂單建立] ──────────▶ [待付款]
                         │
              ┌──────────┼──────────┐
              ▼          ▼          ▼
          [付款中]    [已取消]       │
              │                     │
      ┌───────┴───────┐             │
      ▼               ▼             │
  [已付款]       [付款失敗]──補償──▶[待付款]
      │
      ▼
  [後續履行狀態待 Q5、Q6 確認]
```

> **⚠️ 注意：** 完整的訂單狀態機需待 Q5（庫存）、Q6（退款）、Q8（取消條件）確認後才能定案。

### Feature: 商品目錄管理

```gherkin
Feature: 商品目錄管理
  身為平台管理員與顧客
  我們需要一個可靠的商品目錄
  以便管理員能上下架商品，顧客能瀏覽並選購商品

  Background:
    Given 系統中已存在以下商品：
      | 商品名稱   | 售價（TWD） | 狀態 |
      | 無線滑鼠   | 590         | 上架 |
      | 機械鍵盤   | 2,490       | 上架 |
      | 舊款耳機   | 999         | 下架 |
      | 新品音箱   | 3,200       | 草稿 |

  Scenario: 訪客瀏覽商品列表只看到上架商品
    Given 我是一位未登入的訪客
    When 我瀏覽商品列表
    Then 我應該看到以下商品：
      | 商品名稱 | 售價（TWD） |
      | 無線滑鼠 | 590         |
      | 機械鍵盤 | 2,490       |
    And 我不應該看到「舊款耳機」
    And 我不應該看到「新品音箱」

  Scenario: 管理員成功上架新商品
    Given 我是一位已登入的管理員
    When 我新增一個商品，資料如下：
      | 欄位     | 值              |
      | 商品名稱 | 藍牙喇叭        |
      | 售價     | TWD 1,800       |
      | 描述     | 高音質藍牙喇叭  |
    And 我執行「上架」操作
    Then 商品「藍牙喇叭」的狀態應為「上架」
    And 訪客瀏覽商品列表時應能看到「藍牙喇叭」，售價為 TWD 1,800

  Scenario: 管理員無法建立售價為零的商品
    Given 我是一位已登入的管理員
    When 我嘗試新增一個商品，售價為 TWD 0
    Then 系統應拒絕此操作
    And 系統應顯示錯誤訊息「商品售價必須大於 TWD 0」

  Scenario: 管理員下架商品後顧客無法瀏覽
    Given 我是一位已登入的管理員
    And 商品「無線滑鼠」目前狀態為「上架」
    When 我對「無線滑鼠」執行「下架」操作
    Then 商品「無線滑鼠」的狀態應為「下架」
    And 訪客瀏覽商品列表時不應看到「無線滑鼠」
```

### Feature: 訂單建立

```gherkin
Feature: 訂單建立
```

## 事件風暴分析（Event Storming）
> 來源：photo · 便利貼 21 張 · 辨識出 **3 個 bounded context**、12 個 domain event（顏色語彙依 eventstorming.com：橘=Event、藍=Command、黃=Actor/Aggregate、粉=External、紫=Policy、綠=Read Model、紅=Hotspot）

### Bounded Context：商品數量
| 元素 | 內容 |
|------|------|
| Domain Events（橘） | 商品數量已查詢、商品數量已回覆、庫存數量已取得 |
| Commands（藍） | 扣款 |
| External Systems（粉） | 外部系統 信用卡金流閘道 |

### Bounded Context：採購
| 元素 | 內容 |
|------|------|
| Domain Events（橘） | 採購已請求、金額已預估、庫存已確認可賣、款項已扣款、採購已完成 |
| Commands（藍） | 下單採購、預估金額、確認庫存 |
| Policies（紫） | Policy 可賣則扣款 |

### Bounded Context：退貨
| 元素 | 內容 |
|------|------|
| Domain Events（橘） | 退貨已請求、信用卡已退款、退貨庫存已增加、退貨已完成 |
| Actors（小黃） | 消費者 Customer |
| Policies（紫） | Policy 預估優先、Policy 預估成功驗庫存 |

### 事件流（時間軸）
`商品數量已查詢` → `商品數量已回覆` → `庫存數量已取得` → `採購已請求` → `金額已預估` → `庫存已確認可賣` → `款項已扣款` → `採購已完成` → `退貨已請求` → `信用卡已退款` → `退貨庫存已增加` → `退貨已完成`
